# 计划执行 Agent 指南

这个目录用于存放 Codex 或其他 AI agent 可以按受控顺序执行的计划。

计划文件名应能按预期执行顺序排序，例如：

```text
2026-06-26-001-fix-run-logs.md
2026-06-26-002-verify-storage-usage.md
```

## 目录结构

第一层目录表示计划状态：

- `inbox/`：原始想法或仍需审核的计划草稿。
- `ready/`：已经审核、等待执行的计划。
- `running/`：当前正在执行的计划。
- `blocked/`：因为缺少用户输入或外部依赖而无法继续的计划。
- `done/`：已经完成的计划。
- `failed/`：执行失败且未恢复的计划，需要重新决策。
- `templates/`：可复用的计划模板。

每个第一层目录下按计划类型分组：

- `design/`：产品、交互、页面、流程设计计划。
- `architecture/`：系统架构、模块边界、接口方案、技术选型计划。
- `development/`：功能开发、重构、代码实现计划。
- `testing/`：单测、集成测试、smoke 测试、验证计划。
- `deployment/`：构建、发布、远端部署、回滚、上线验证计划。

## 计划文件要求

每个计划都应足够自包含，让新的 agent 会话可以直接执行：

- 目标和预期结果
- 相关文件、服务、主机或 URL
- 明确的执行步骤顺序
- 需要运行的验证命令或需要保留的证据
- 适用时的回滚条件或停止条件

## Agent 执行规则

- 除非用户指定其他子目录或文件，否则从 `ready/` 开始处理。
- 递归扫描 `ready/` 下所有类型目录，并按文件名全局排序执行，不按类型优先级排序。
- 一次只执行一个计划。
- 执行中的计划应移动到 `running/`，并保留原类型目录。
- 完成的计划应移动到 `done/`，并在计划文件中保留最终说明或验证证据。
- 被阻塞的计划应移动到 `blocked/`，并在计划文件中记录阻塞原因。
- 失败的计划应移动到 `failed/`，并在计划文件中记录错误和最后确认的状态。
- 在状态目录之间移动计划时必须保留类型路径，例如 `ready/development/foo.md` 完成后移动到 `done/development/foo.md`。
- 除非用户明确要求，否则不要调整计划顺序。
